## Quality Review Findings

### [Critical] Finding 1: Concurrency Lost Update Race Condition in DB Helper

- **What**: Concurrent read-modify-write operations on the JSON database suffer from lost updates and data truncation.
- **Where**: `lib/db/db.ts` (Promise queue serialization implementation).
- **Why**: The promise queue wrapper `enqueue` serializes individual reads and writes separately. However, when concurrent updates are made (e.g., in `Test Case 4.4` and `Test Case 2.6.2`), all `readDb()` tasks are enqueued and executed first. This means every concurrent thread reads the *same stale seed database state* from disk, modifies its own isolated copy, and then enqueues a `writeDb()` task. When the writes run sequentially, each write completely overwrites the file with its own isolated copy, discarding the results of all preceding writes. At the end of 100 concurrent writes, only 1 branch remains in the database (plus the seed branch), rather than 100.
- **Suggestion**: Implement a serialized transaction helper, such as:
  ```typescript
  export async function updateDb(updater: (db: DatabaseSchema) => void | Promise<void>): Promise<void> {
    return enqueue(async () => {
      const db = await readDb(); // Must read inside the enqueued scope
      await updater(db);
      await writeDb(db);
    });
  }
  ```
  Alternatively, maintain a shared in-memory database cache reference so concurrent mutations are applied to the same object before writing.

### [Minor] Finding 2: `validateSchema` Type Safety and Robustness Gap

- **What**: The `validateSchema` function does not verify or default the `mergeProposals` key.
- **Where**: `lib/db/db.ts` (lines 31-60).
- **Why**: The `DatabaseSchema` interface defines `mergeProposals: MergeProposal[]` as a required field. However, `validateSchema` does not assert `Array.isArray(db.mergeProposals)`. If an older JSON database or one edited manually lacks this field, it will pass schema validation, but any subsequent Next.js server actions or DB helpers attempting to access or modify `db.mergeProposals` (e.g., `db.mergeProposals.push(...)`) will crash with a runtime TypeError.
- **Suggestion**: Add a schema validation check:
  ```typescript
  if (!Array.isArray(db.mergeProposals)) return false;
  ```

### [Minor] Finding 3: Missing `NODE_ENV` Configuration in Test Runner

- **What**: The test runner fails to set `process.env.NODE_ENV = 'test'`.
- **Where**: `scripts/run-e2e.js`.
- **Why**: `TEST_INFRA.md` specifies that the E2E test runner must set `process.env.NODE_ENV = 'test'` to ensure environment isolation. Currently, it only configures `MEMFORK_DB_PATH`.
- **Suggestion**: Add `process.env.NODE_ENV = 'test';` to the environment sandbox setup in `scripts/run-e2e.js`.

## Adversarial Critic Findings

### [High] Finding 4: Fact Parsing Failure for Bullet-Point/Hyphen Facts

- **What**: Facts starting with a hyphen (such as bullet points) are ignored and omitted from commits in the Mock CLI.
- **Where**: `scripts/mock-memfork.js` (lines 127-130).
- **Why**: The mock argument parser loops through facts and breaks early if it encounters any argument starting with a hyphen to avoid capturing trailing flags:
  ```javascript
  if (args[i].startsWith('-')) break;
  ```
  However, facts generated by LLMs/agents often begin with hyphens (e.g., `"- Neo4j is a graph database"`). In `process.argv`, these strings start with `-` regardless of shell quoting. As a result, the parser terminates fact capture immediately, causing the mock CLI to save an empty facts list `[]`.
- **Suggestion**: Since the wrapper `memfork.ts` always appends `--facts` as the last parameter, the mock parser can safely capture all remaining arguments as facts, or check them against a specific blacklist of known CLI options (e.g., `-m`).

### [Medium] Finding 5: Infinite Cycle Hang Hazard in Branch Recall

- **What**: Cycle in branch lineage leads to infinite loop in `recall`.
- **Where**: `scripts/mock-memfork.js` (lines 178-181).
- **Why**: The branch lineage traversal uses a simple `while (current)` loop matching `parentBranchId`. If a circular dependency is introduced into the JSON DB (e.g., `branchA.parentBranchId = 'branchB'` and `branchB.parentBranchId = 'branchA'`), the recall command will hang indefinitely until it hits the 10-second child process timeout.
- **Suggestion**: Keep track of visited branch IDs during traversal and break if a cycle is detected.
